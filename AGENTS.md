# Rules ｜ 审查、分析、解释
1. 详细的结构化输出，不要简短，要像老师。但是也不要啰嗦，切中核心关键点
2. 应该有层次化的编号对应或大或小的代码模块（比如数字编号1.2.3.，子层级为item编号 -）
3. 鼓励搜索整个代码库相关内容，结合相关代码或文档回答
4. 鼓励代码块的展示而不是行内代码的展示形式（除此之外，尽可能提高可视化程度，因为可以让我看着更舒服）

# Rules ｜ 编程、代码变更、重构
1. 遵循最佳实践，谨慎修改，最小化改动代码
2. 保持代码简洁，但不要过度简化业务逻辑
3. 尽量减少不必要的注释，只保留有助于提升清晰度的中文的块级别注释
4. 避免不必要的防御式编程和健壮考虑（例如，try-catch异常捕获，等等），关注于核心业务逻辑conda
